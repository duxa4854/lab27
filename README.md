## Ответы на вопросы

### 1. Чем Minimal API отличается от MVC в ASP.NET Core?
*   **Minimal API** предназначено для создания легких HTTP API с минимальным количеством зависимостей и файлов. Оно идеально подходит для микросервисов и простых приложений, где не нужна сложная структура. Весь код часто находится в одном файле `Program.cs`.
*   **MVC (Model-View-Controller)** — это полноценный фреймворк с строгой архитектурой, разделением ответственности (Контроллеры, Модели, Представления), встроенной поддержкой Razor Pages, фильтрами, сложной маршрутизацией через атрибуты и т.д. Он лучше подходит для больших монолитных приложений и проектов со сложной бизнес-логикой и UI.

### 2. Что произойдёт, если поставить `app.MapGet(...)` до `app.Use(...)`?
Порядок регистрации middleware в ASP.NET Core имеет критическое значение. Middleware выполняется в том порядке, в котором оно добавлено в конвейер (pipeline).
*   Если вы зарегистрируете конечную точку (`MapGet`) **до** middleware (`Use`), то запрос к этому маршруту **минует** данное middleware.
*   Middleware, зарегистрированное после `MapGet`, не будет выполнено для этого конкретного запроса, так как обработка завершится на этапе маппинга маршрута (если внутри handler'а не вызывается следующий компонент явно, но в случае терминальных операций типа `MapGet` конвейер обычно завершается).
*   *Исключение:* Если middleware регистрируется через `app.UseEndpoints` или аналогичные механизмы в старых версиях, но в Minimal API `MapGet` сам является частью конвейера. Главное правило: **глобальные middleware (логирование, аутентификация, обработка ошибок) должны идти перед маппингом маршрутов.**

### 3. Почему ASP.NET Core преобразует PascalCase в camelCase при сериализации JSON?
Это сделано для соблюдения соглашений именования разных языков:
*   В **C#** общепринятым стандартом для имен свойств классов является **PascalCase** (например, `UserName`).
*   В **JavaScript/JSON** (и большинстве веб-стандартов) стандартом является **camelCase** (например, `userName`).
*   ASP.NET Core по умолчанию использует политику именования `JsonNamingPolicy.CamelCase` в сериализаторе `System.Text.Json`. Это обеспечивает бесшовную интеграцию с фронтенд-фреймворками (React, Angular, Vue), которые ожидают данные в формате camelCase, избавляя разработчиков от необходимости вручную переименовывать поля на клиенте или сервере [[38]], [[40]].

### 4. Что означает код ответа HTTP 401? А 404? А 200?
*   **200 OK**: Запрос успешно выполнен. Сервер вернул ожидаемые данные.
*   **401 Unauthorized**: Запрос требует аутентификации пользователя. Клиент не предоставил действительные учетные данные (или не предоставил их вовсе). Это не то же самое, что 403 (Forbidden), который означает, что сервер понял запрос, но отказывает в доступе даже при наличии аутентификации.
*   **404 Not Found**: Сервер не может найти запрашиваемый ресурс. Обычно это означает, что URL неверен или ресурс был удален.

### 5. Чем `dotnet run` отличается от `dotnet watch run`?
*   **`dotnet run`**: Компилирует и запускает приложение один раз. Если вы внесете изменения в код, вам нужно остановить приложение (Ctrl+C), перекомпилировать и запустить его снова, чтобы увидеть изменения.
*   **`dotnet watch run`**: Запускает приложение и отслеживает изменения в исходных файлах. Как только вы сохраняете измененный файл, инструмент автоматически перекомпилирует проект и перезапускает приложение. Это значительно ускоряет процесс разработки (Hot Reload), так как не требует ручного перезапуска [[1]].
